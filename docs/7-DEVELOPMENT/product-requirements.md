# AI 小说转剧本工具 PRD

## 1. 产品概述

### 1.1 产品名称

AI 小说转剧本工具

### 1.2 产品愿景

让小说作者能够在几分钟内，把连续多章小说正文转换成一份结构化、可编辑、可继续精修的剧本初稿，把“从长篇叙事到戏剧表达”的改编门槛从专业编剧能力，降低到可由作者和编辑协作完成的工作流。

### 1.3 核心价值

- 降低改编门槛：作者不必从空白页开始写剧本。
- 提升改编效率：自动完成章节理解、分场、对白化和结构化输出。
- 保持可控性：所有输出都带来源追溯和改编说明，便于人工核对与打磨。

## 2. 问题定义

### 2.1 用户痛点

- 小说作者擅长 prose 写作，但不熟悉剧本格式和分场逻辑。
- 改编成剧本时，需要手动抽取事件、压缩描写、重组冲突，工作量大。
- 现有通用 AI 输出往往是自由文本，不稳定、不可追溯、难编辑。
- 编辑和策划需要一个可统一审阅的中间格式，而不是一次性最终成稿。

### 2.2 我们要解决的问题

系统需要把同一作品连续 3 个及以上章节的小说文本，转成稳定的 YAML 剧本结构，既能被作者直接看懂修改，也能被后续编辑器、导出器和自动化校验程序继续处理。

## 3. 目标用户

### 3.1 核心用户

- 网络小说作者
- IP 改编工作室编辑
- 短剧/网剧内容策划

### 3.2 次级用户

- 编剧助理
- 下游工具开发者
- 内容平台内部运营和审校人员

## 4. 使用场景

### 4.1 核心场景

作者选择作品中的第 12 到 15 章，设置“短剧风格、节奏优先、18 分钟时长”，系统输出一份 YAML 剧本初稿，作者随后逐场修改对白和节奏。

### 4.2 扩展场景

- 编辑批量审查不同改编策略的输出差异。
- 团队将 YAML 接入内部编辑器，继续补完镜头、分集和版本管理。
- 平台基于 YAML 做角色出场分析、节奏分析和导出排版。

## 5. 产品目标

## 5.1 商业目标

- 验证“小说作者愿意为 AI 改编辅助工具付费”的需求。
- 建立从小说文本到结构化剧本的标准数据层。
- 为后续扩展到富文本编辑器、导出器和协作审校打基础。

### 5.2 用户目标

- 在 5 分钟内得到一份首版剧本初稿。
- 能明确知道每场戏来自哪些章节。
- 能方便地修改局部对白、动作和桥接逻辑。

### 5.3 产品成功指标

- 首次生成成功率 >= 90%
- Schema 校验通过率 >= 95%
- 用户平均首次可编辑时间 <= 5 分钟
- 用户人工保留率：>= 60% 场景被保留而非完全重写
- 单次生成后继续编辑率 >= 50%

## 6. MVP 范围

### 6.1 In Scope

- 导入连续 3 到 10 章中文小说文本
- 支持作品标题、章节标题、目标格式、目标风格、目标时长输入
- 自动生成 YAML 剧本初稿
- 输出角色、地点、道具、场景和改编说明
- 输出章节来源映射和质量提示
- 支持 Schema 校验

### 6.2 Out of Scope

- 最终拍摄级导演脚本
- 分镜头脚本
- 多人实时协作
- 自动版权判定
- 一次性处理整本超长小说
- Final Draft/Fountain 正式导出器

## 7. 功能需求

### 7.1 输入与导入

#### 功能描述

用户可以粘贴或上传连续章节文本，并填写最少的辅助信息。

#### 需求点

- 支持输入作品标题
- 支持输入章节号和章节标题
- 支持检测章节数量是否 >= 3
- 支持检测章节顺序是否连续
- 支持设置目标格式
- 支持设置改编模式
- 支持设置目标时长

#### 验收标准

- 当章节数少于 3 时，系统阻止生成并给出明确提示
- 当章节号不连续时，系统阻止生成并提示缺失章节
- 当必填字段缺失时，系统阻止生成

### 7.2 内容理解与改编规划

#### 功能描述

系统在正式输出 YAML 前，需要完成结构化理解和改编决策。

#### 需求点

- 抽取角色、地点、关键事件、核心冲突
- 识别不适合直接搬运到剧本的长篇叙述
- 输出改编策略摘要
- 标记保留主线、删减支线和桥接内容

#### 验收标准

- 输出中必须包含 `adaptation.strategy_summary`
- 输出中每个场景必须包含 `source_refs`
- 如存在桥接场景，应在 `adaptation_notes` 或 `adaptation.added_bridges` 中显式说明

### 7.3 YAML 剧本生成

#### 功能描述

系统生成符合正式 Schema 的 YAML 剧本初稿。

#### 需求点

- 支持稳定输出 `script/source/adaptation/characters/locations/props/scenes/quality_notes`
- 支持场景头、场景摘要、出场角色、内容块
- 支持动作、对白、转场等内容块类型
- 支持整体质量提示和置信度

#### 验收标准

- 输出必须通过 JSON Schema 校验
- 每个场景必须至少有 1 个正文块
- 对白块必须带 `character_id`
- 生成失败时要返回明确错误和失败阶段

### 7.4 编辑与二次生成

#### 功能描述

用户在拿到初稿后，可以继续调整。

#### 需求点

- 支持按场景查看和编辑 YAML
- 支持按角色、章节来源筛选场景
- 支持局部重生成单场或单段内容
- 支持保留人工修改痕迹

#### 验收标准

- 用户可以只修改某个场景而不影响其他场景
- 用户可以只重生成某场对白
- 系统必须保留 schema_version 和稳定 ID

## 8. 非功能需求

### 8.1 性能

- 3 到 10 章输入在正常负载下应于 60 秒内完成首次生成
- 单场局部重生成应于 20 秒内完成

### 8.2 可用性

- 错误提示应明确指出是输入问题、理解问题、生成问题还是校验问题
- 输出应保证作者无需懂 JSON/YAML 规范也能阅读

### 8.3 可靠性

- 生成结果需要自动做结构校验
- 如果模型输出非法结构，系统必须自动修复或返回可理解错误

### 8.4 安全与合规

- 用户作品必须按租户隔离
- 不得将用户作品默认用于模型训练
- 需允许用户删除上传原文和生成结果

## 9. 用户故事

### 9.1 作者

作为小说作者，我希望上传几章正文后立刻拿到一个剧本初稿，这样我不用从零学习剧本格式也能开始改编。

### 9.2 编辑

作为编辑，我希望看到每场戏对应的来源章节和改编说明，这样我能快速判断改编是否忠于原作、删改是否合理。

### 9.3 工具开发者

作为下游工具开发者，我希望输出结构稳定且带版本号，这样我能安全地接入编辑器、导出器和校验器。

## 10. 核心流程

1. 用户创建改编任务
2. 输入作品与章节信息
3. 系统完成章节校验
4. 系统执行内容理解与改编规划
5. 系统生成 YAML 初稿
6. 系统执行 Schema 校验
7. 用户查看、编辑、局部重生成
8. 用户导出或继续加工

## 11. 风险与应对

### 11.1 内容理解偏差

- 风险：模型错误理解角色关系或事件因果
- 应对：保留 `source_refs`、`open_questions`、质量提示和人工可编辑入口

### 11.2 结构输出不稳定

- 风险：模型输出缺字段、错类型或层级错乱
- 应对：强制 Schema 校验 + 自动修复链路

### 11.3 用户期望过高

- 风险：用户把“初稿工具”当成“终稿编剧”
- 应对：产品文案中明确“AI 辅助初稿”定位

### 11.4 版权与数据敏感性

- 风险：用户担心原文泄露和训练用途
- 应对：显式数据策略、删除机制和租户隔离

## 12. 版本规划

### Phase 1

- 多章节导入
- YAML 初稿生成
- JSON Schema 校验
- 基础编辑视图

### Phase 2

- 局部重生成
- 角色/场景筛选
- 版本对比
- 多种改编模式对照输出

### Phase 3

- 分集结构
- Fountain/Final Draft 导出
- 协作审校
- 节奏和商业钩子分析

## 13. 发布准入标准

- 关键生成链路完成端到端走通
- 示例 YAML 全部通过 Schema 校验
- 错误处理覆盖输入、生成、校验三类失败
- 至少完成 3 个不同题材样本的人工评审
