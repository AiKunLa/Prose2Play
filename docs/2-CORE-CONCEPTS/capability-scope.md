# AI 小说转剧本工具能力定义

## CAPABILITY

面向小说作者、工作室编辑和内容策划，提供一项“将连续 3 个及以上章节的小说文本转换为结构化剧本初稿”的能力。用户上传或粘贴原始小说章节后，系统自动完成章节理解、人物和场景抽取、剧情压缩、戏剧节奏重组，并输出可编辑的 YAML 剧本草稿，帮助作者把“长文本叙事”快速变成“分场、分角色、可继续打磨的剧本骨架”。

## CONSTRAINTS

### 固定业务规则

- 输入必须是同一作品内连续的 3 个及以上章节。
- 输出必须是结构化 YAML，而不是自由文本剧本。
- 输出目标是“可编辑初稿”，不是终稿，不承诺一次生成即可拍摄。
- 每个场景都必须保留与原小说的追溯关系，至少能回溯到章节级别。
- 系统需要显式区分“忠实改编内容”和“为戏剧化补足而新增的桥接内容”。

### 范围边界

- 本期聚焦“小说到剧本初稿”的转换，不负责直接生成分镜、镜头表、预算表或拍摄计划。
- 本期默认产出单集或单段剧情脚本，不覆盖整部长篇小说的一次性全量改编。
- 本期优先支持中文输入和中文输出。
- 本期不解决版权归属，只保留版权声明和来源追踪字段。

### 不可破坏的产出约束

- YAML 结构必须稳定，便于后续被前端编辑器、版本比对工具和导出器消费。
- 输出需要同时满足“模型容易生成”和“人类容易阅读编辑”。
- 每个场景都应具备最小剧本单元：场次编号、场景标题、时间、地点、出场人物、剧情摘要、正文内容。
- 正文内容必须按可渲染块组织，例如动作、对白、提示语、转场。

### 质量约束

- 生成结果不能只做摘抄，需要完成叙事压缩与戏剧化重组。
- 不能凭空改变主线人物关系和关键事件因果，除非在改编说明中显式标记。
- 对跨章节信息要做人物一致性校验，避免名称漂移、身份冲突和时序错误。
- 当模型对原文理解不确定时，应以 `open_questions` 或 `adaptation_notes` 显式暴露，而不是静默胡编。

## IMPLEMENTATION CONTRACT

### 目标用户

- 小说作者：希望快速看到作品的剧本化版本。
- 编辑/策划：希望用统一结构审阅故事节奏、角色出场和分场组织。
- 后续工具链：YAML 编辑器、剧本渲染器、版本比对器、协作审校系统。

### 关键用户流程

1. 用户输入连续 3 个及以上章节的小说正文。
2. 系统进行章节切分、人物识别、地点识别、冲突提取和剧情摘要。
3. 系统生成“改编方案”，确定保留主线、合并事件、拆分场景和桥接逻辑。
4. 系统输出结构化 YAML 剧本草稿。
5. 用户在编辑器中修改场景、对白、节奏和改编说明。
6. 系统支持二次生成或导出为其他格式。

### 核心能力拆解

#### 1. 输入整理层

- 接收多章节正文、章节标题、章节顺序。
- 校验章节是否连续、是否属于同一作品。
- 支持用户补充题材、目标风格、目标时长、改编偏好。

#### 2. 叙事理解层

- 抽取角色、地点、关键事件、冲突、情绪转折。
- 建立章节摘要和章节间因果链。
- 标记不适合直接舞台化或影视化的内心独白、世界观说明和叙述性段落。

#### 3. 改编规划层

- 将小说事件重排为剧本场景。
- 决定哪些内容保留为动作，哪些转换为对白，哪些需要桥接场景。
- 控制节奏密度，避免长篇叙述直接塞入单场。

#### 4. 结构化输出层

- 生成符合 Schema 的 YAML。
- 给每个场景附带来源章节映射。
- 记录改编说明、待确认问题和机器置信不足点。

#### 5. 编辑与迭代层

- 支持按场景、角色、章节来源进行局部重写。
- 支持只重生成某一场、某一角色对白或某段桥接内容。
- 保留 schema_version 以支持未来升级。

### 推荐 MVP

#### MVP 用户承诺

- 可将 3 到 10 章中文小说文本转换为 8 到 30 场左右的 YAML 剧本初稿。
- 输出包含角色表、场景表、正文块、章节映射和改编说明。
- 用户可以在不懂编程的情况下看懂并编辑 YAML。

#### MVP 输入

- 作品标题
- 章节列表
- 目标体裁，例如短剧、网剧、电影大纲式剧本
- 目标风格，例如忠实改编、节奏优先、对白强化

#### MVP 输出

- 一份 YAML 剧本文件
- 一份生成摘要，说明改编策略、压缩比、删改重点和待确认问题

### 接口与数据含义

- 输入接口的最小粒度是章节。
- 输出接口的最小粒度是场景内内容块。
- 角色、地点、道具使用稳定 ID，避免名称修改后引用断裂。
- 每个场景需要包含 `source_refs`，支持从场景追到章节。

### 状态与生命周期

- `ingested`：原文已导入并校验。
- `analyzed`：章节理解完成。
- `planned`：改编规划完成。
- `drafted`：YAML 初稿已生成。
- `edited`：用户已修改。
- `exported`：导出到下游格式。

### 安全与策略

- 对用户上传文本按作品隔离存储。
- 如果提供云端服务，需要明确版权声明、数据留存周期和模型训练策略。
- 对敏感题材或违规内容要有审核与拒绝策略。

### 可观测性

- 记录章节数、字数、生成耗时、场景数、角色数、桥接场景比例。
- 记录结构化校验失败位置，便于修复模型输出。
- 记录用户最常编辑的字段，反推 Schema 和生成策略优化方向。

## NON-GOALS

- 不负责替代专业编剧完成终稿润色。
- 不在 MVP 中生成镜头语言级别的导演脚本。
- 不在 MVP 中做多人协作审校工作流。
- 不在 MVP 中做自动版权确权或侵权判断。
- 不在 MVP 中处理整本超长小说的一键全集改编。

## OPEN QUESTIONS

- 首发更偏“影视短剧”还是“舞台剧/广播剧”？
- 场景正文是否需要进一步细分为镜头级块，还是保持剧本级块即可？
- 是否允许用户指定改编压缩比例，例如 3 章压成 5 场或 12 场？
- 是否需要为每个场景输出情绪标签、商业钩子标签和短视频卡点标签？
- 长期是否要支持从 YAML 一键导出 Final Draft、Markdown 或 Fountain？

## HANDOFF

当前已经具备直接进入原型设计和实现拆解的条件。下一步建议按以下顺序推进：

1. 基于 Schema 做一个 YAML 样例库和校验器。
2. 设计“章节导入 -> 生成 -> 编辑”的最小交互流。
3. 进入 `tdd-workflow`，先为 YAML 校验、章节连续性校验和场景映射写测试。
4. 进入 `verification-loop`，验证不同题材小说下的输出稳定性和可编辑性。
